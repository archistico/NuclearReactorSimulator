# M8.3 — Instrumentation & Control Faults

**Status:** BASELINE CANDIDATE

## Scope

- concrete deterministic sensor-fault applicators over the validated M5.1 `SensorFaultInput` seam: bias, freeze, failed-low, failed-high and unavailable;
- deterministic controller-output freeze/fail-low/fail-high effects implemented as temporary per-step command-input overrides over canonical M5.2–M5.4 controller definitions;
- deterministic actuator-command freeze/fail-low/fail-high effects for unambiguously one-controller/one-actuator command paths;
- exact target validation against canonical instrumentation channel, controller and actuator IDs;
- automatic built-in M8.3 applicator registration alongside the validated M8.2 hydraulic fault family;
- deterministic demonstration and protection fail-safe diagnostic scenario definitions;
- protection/interlock behavior remains owned by M5.5 and consumes the same committed faulted `MeasuredSignalFrame`; M8.3 does not inject protection outcomes directly.

## Acceptance

- sensor bias changes only the measured signal and reports degraded quality through M5.1; true state is never rewritten;
- freeze holds the committed prior output using existing M5.1 instrumentation state and marks the measurement invalid/suspect;
- failed-low/high and unavailable reuse the exact M5.1 validity/quality semantics;
- a faulted protection channel affects M5.5 only through committed measured-frame ordering, including existing fail-safe invalid-measurement policy;
- controller-output faults force only canonical controller command output through bounded manual-mode input substitution; they do not write physical pump/valve/rod state directly;
- actuator-command faults are accepted only when the target actuator maps unambiguously to a single controller command path; ambiguous shared-controller mappings fail closed;
- clearing a control fault removes only the temporary forcing override so the latest persistent operator/controller input resumes through normal bumpless/manual-auto semantics;
- unknown channel/controller/actuator IDs, conflicting simultaneous faults on the same channel/controller and invalid bias parameters fail closed;
- no wall-clock/random behavior or true-state fallback is introduced;
- complete build and automated suite must pass locally before M8.3 validation.

## Deliberate limits

M8.3 does not model hidden electronics, stochastic noise, calibration drift as a new physical process, relay hardware internals or arbitrary protection-logic corruption. Sensor drift can be represented later by deterministic scheduled bias changes if needed, but M8.3 does not add a second time integrator for it. Protection/interlock logic itself remains M5.5 ownership: diagnostic scenarios perturb measured inputs or command paths and observe the resulting canonical response. Turbine/generator/feedwater/condenser transient packs remain M8.4 ownership.
