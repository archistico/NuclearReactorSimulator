# M10 — Operator Computer, Supervisory Automation & Human-Machine Integration

## Status

**IN PROGRESS — M10.1–M10.6 VALIDATED; M10.7 IMPLEMENTATION CANDIDATE.**

## Goal

Create a unified fixed-page operator terminal over existing canonical control-room/training/session capabilities and add a separately owned deterministic supervisory-automation layer under M5.

## Approved sequence

- M10.1 Operator Computer Contracts & Terminal Shell — **VALIDATED**
- M10.2 Unified Information, Guidance & Diagnostics — **VALIDATED**
- M10.3 Alarm, Log & Incident Workstation — **VALIDATED**
- M10.4 Contextual Command Console — **VALIDATED**
- M10.5 Dual Assistance & Control-Authority Model — **VALIDATED**
- M10.6 Supervisory Automatic Operation — **VALIDATED**
- M10.7 Session, Checkpoint, Replay & Save Workspace — **IMPLEMENTATION CANDIDATE**
- M10.8 Integrated Operator Computer UI
- M10.9 Integrated Human-Automation Validation Gate

## Architectural invariants

- the computer aggregates; it does not own physics/control/alarm/guidance/session truth;
- training assistance and physical control authority are independent axes;
- real supervisory automation is M5-owned and acts through existing controller/setpoint/command seams;
- protection/interlocks always have priority;
- measured-signal discipline, fail-closed degradation and bumpless manual takeover are mandatory;
- fixed menu/pages only; no free-form textual command parser;
- save/load remains exact-version and replay-backed through M9.1 ownership;
- keyboard-only operation is a phase acceptance requirement.
