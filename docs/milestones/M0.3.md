# M0.3 — Simulation Test Harness & Runtime Hardening

## Goal

Finish the pre-physics simulation foundation so M1 can introduce physical quantities and plant primitives on top of deterministic, fault-aware and deeply testable runtime contracts.

## Scope

### Runtime hardening

- terminal `Faulted` runtime state;
- stable immutable `SimulationFaultSnapshot` diagnostics;
- `SimulationRuntimeFaultException` preserving the original failure as inner exception;
- transactional step commit semantics;
- failed-step command restoration in original FIFO order;
- rejection of further runtime mutation/execution after a fault;
- explicit immutable/copy-on-write state ownership requirement for future plant states.

### Invariant infrastructure

- `ISimulationInvariant<TState>`;
- deterministic invariant evaluation after kernel calculation and before commit;
- structured `SimulationInvariantResult`;
- `SimulationInvariantViolationException` with invariant name, reason and failed step.

### Command trace and replay primitives

- immutable `SimulationCommandTrace<TCommand>`;
- logical step-indexed `SimulationCommandTraceEntry<TCommand>`;
- deterministic same-step ordering;
- headless `SimulationReplayRunner` using fixed single-step execution;
- no wall-clock timestamps and no replay serialization in the Simulation layer.

### Reusable test harness

- generic deterministic scenario harness in `Simulation.Tests`;
- logical-step command scheduling;
- capture of initial and every committed step snapshot;
- reusable final/intermediate snapshot assertions for future physical scenarios.

### Stress and long-run verification

- 100,000-step pulse-segmentation equivalence test;
- large deterministic command-trace replay verification;
- kernel-fault rollback tests;
- invariant-fault rollback tests;
- multi-step partial-commit verification.

## Transactional step contract

```text
last committed state
        │
        ▼
create step context
        │
        ▼
drain FIFO commands
        │
        ▼
calculate candidate state
        │
        ▼
evaluate invariants
        │
create candidate snapshot
        │
   ┌────┴────┐
   │ success │ failure
   ▼         ▼
commit     restore commands
clock      keep old state/time
state      mark Faulted
   │         │
   ▼         ▼
next step  diagnostics only
```

A physical kernel must treat the input state as immutable/copy-on-write. The runtime cannot roll back mutations performed directly inside an object already referenced as committed state.

## Deterministic trace contract

Given the same:

```text
initial state
fixed timestep
logical command trace
final logical step
```

a replay must produce the same final snapshot and the same sequence of intermediate snapshots.

## Explicitly out of scope

- reactor/neutron physics;
- fluids, pressure or thermodynamics;
- persistence and serialized replay files;
- checkpoints and recovery of a faulted runtime;
- background scheduler/thread;
- production scenario authoring UI;
- control-room UI.

## Acceptance checklist

Run from the repository root:

```powershell
dotnet restore
dotnet build --no-restore
dotnet test --no-build
```

Expected validation:

1. restore succeeds;
2. build succeeds with zero warnings and zero errors;
3. all tests pass;
4. Avalonia application starts and identifies M0.3;
5. a kernel failure leaves state/time at the last committed step and marks the runtime `Faulted`;
6. commands drained for a failed step remain pending in original order;
7. invariant violations prevent candidate-state commit;
8. faulted runtimes reject further execution and mutation while remaining inspectable;
9. identical logical command traces replay identically;
10. scenario harness captures initial and per-step snapshots deterministically;
11. the 100,000-step segmentation stress test passes;
12. architecture guards against wall-clock/timer APIs remain green.
