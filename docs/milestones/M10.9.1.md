# M10.9.1 — HMI Information Architecture & Visual Language

**Status:** VALIDATED  
**Prerequisite baseline:** M10.8 — Integrated Operator Computer UI — VALIDATED

## Goal

Establish the structural and semantic foundation for the complete operator-experience refactor before advanced gauges, plant mimics, command-consequence visualization or challenge scoring are added.

M10.9.1 changes how information is organized and interpreted. It does **not** add new plant physics, control logic, protection logic, fault logic, training scoring semantics or automation authority.

## Operator-experience shell

The desktop is reorganized into five persistent regions:

1. **Situation strip** — runtime, logical step, gross electrical output, current training score, unacknowledged alarms, protection state, assistance mode and effective control authority.
2. **Compact system rail** — PLANT, REACTOR, PRIMARY, TURBINE, GRID, ALARMS and COMPUTER.
3. **Central workspace** — existing validated subsystem/operator-computer content remains the primary detailed work area.
4. **Context inspector** — current condition, next canonical action, subsystem context and latest command feedback remain visible beside the workspace.
5. **Alarm/event strip** — unacknowledged alarm count, first-out status, latest deterministic event and runtime step remain visible at the bottom edge.

The shell deliberately reserves the center for the future M10.9.3/M10.9.4 plant and subsystem schematics instead of continuing to expand navigation/card chrome.

## Visual-language rules

- dark industrial/futuristic HMI rather than decorative cyberpunk styling;
- information cyan/ice accents are distinct from state colors;
- green means confirmed healthy/available;
- amber means attention/warning;
- red is reserved for alarm/trip/protection significance;
- unavailable information stays unavailable;
- `MEASURED` and `MODEL` provenance remain explicit;
- internal milestone/ownership labels are removed from primary operator-facing headings where they do not help operation;
- progressive disclosure: headline condition first, detailed engineering context on selection/drill-down.

## Instrument semantics contract

M10.9.1 introduces immutable Application HMI scale metadata so later gauge controls never infer safety merely from a value being inside the displayed scale.

The following concepts are distinct:

- **instrument scale** — minimum/maximum that can be displayed;
- **normal/warning/alarm operating bands** — operational interpretation;
- **target band** — scenario/controller objective;
- **setpoint** — requested controller/reference value;
- **protection limits** — published threshold markers owned by the protection domain.

Avalonia may render these semantics but must never invent thresholds or recompute protection logic.

## Explicit non-scope

Deferred to later approved milestones:

- M10.9.2 advanced linear/circular gauges;
- M10.9.3 interactive full-plant mimic;
- M10.9.4 subsystem engineering schematics;
- M10.9.5 command consequence model and highlighted dependency chains;
- M10.9.6 timed challenges, deterministic grid-demand profiles and multidimensional scoring;
- M10.9.7 mission/performance workstation;
- M10.9.8 integrated human-automation-HMI validation gate.

No future grid-demand value is fabricated in M10.9.1 merely to fill the new situation strip.

## Acceptance

Automated coverage must verify:

- the five-region shell contract and compact `188,*,300` body layout;
- situation strip binds only already available presentation/runtime/training/authority data;
- context inspector exposes current condition, next canonical action, subsystem context and command feedback;
- no fake `GRID DEMAND` field is introduced before M10.9.6;
- operator-facing primary headings no longer expose implementation-milestone prefixes;
- HMI scale contracts reject semantic markers outside the display scale and preserve independent scale/band/target/setpoint/protection concepts;
- all pre-existing canonical command bindings, M10.7.1 control feedback and M10.8 F1–F8 operator-computer integration remain intact.

## Validation

Run locally:

```text
dotnet clean
dotnet restore
dotnet build --no-restore
dotnet test --no-build
```

Manual checks:

1. confirm the new shell is clearer at the validated minimum window size;
2. confirm all seven workspaces remain reachable and the selected workspace is obvious;
3. confirm status/alarms/next action remain readable while the center workspace scrolls;
4. confirm F1–F8 still opens the integrated operator computer without plant-state side effects;
5. confirm Run/Pause/Single step/Reset session and all existing operational controls behave exactly as before;
6. confirm no critical information is clipped or hidden by the new 188/300 side regions.

## Validation result

The user explicitly confirmed local compilation and the complete automated test suite passed. M10.9.1 is therefore the official validated baseline for M10.9.2.
