# M0.2 — Deterministic Simulation Runtime

## Goal

Turn the validated M0.1 architecture into a headless, deterministic runtime that can later host reactor and plant physics without coupling physical evolution to Avalonia or the operating-system clock.

## Scope

- deterministic `SimulationClock` with a positive fixed timestep;
- generic headless `SimulationRuntime<TState, TCommand, TStateSnapshot>`;
- pause and resume;
- paused single-step execution;
- exact supported speed multipliers: 0.25×, 0.5×, 1×, 2×, 5× and 10×;
- fixed-point speed scaling without binary floating-point drift;
- thread-safe FIFO command queue with monotonic sequence numbers;
- command consumption only at fixed-step boundaries;
- immutable runtime and simulation snapshot envelopes;
- deterministic pulse-segmentation and replay tests;
- architecture guard preventing wall-clock/timer APIs from entering `Simulation`;
- documentation of command and snapshot ownership rules.

## Runtime contract

```text
External scheduler / test harness
             │
             │ elapsed duration
             ▼
      SimulationRuntime
       │             ▲
       │ fixed steps │ immutable snapshots
       ▼             │
    Plant kernel ────┘
       ▲
       │ FIFO commands consumed only at a step boundary
       │
 Application / future operator command mapping
```

The runtime never asks the operating system what time it is. It is explicitly driven by callers.

## Determinism contract

Given:

```text
same initial state
+ same fixed timestep
+ same ordered command sequence
+ same external-duration/speed sequence
```

then the runtime must produce the same logical step sequence and resulting snapshots.

Different partitioning of the same elapsed duration into UI/scheduler pulses must not alter the physical result.

## Explicitly out of scope

- reactor physics;
- thermodynamics and fluid networks;
- background simulation thread or UI scheduler;
- persistence/save files;
- scenario engine;
- replay serialization;
- production operator-command types;
- control-room UI.

These remain later milestones. M0.2 establishes the runtime contracts they will use.

## Acceptance checklist

Run from the repository root:

```powershell
dotnet restore
dotnet build --no-restore
dotnet test --no-build
```

Expected validation:

1. restore succeeds;
2. build succeeds with zero warnings and zero errors;
3. all tests pass;
4. Avalonia application starts and identifies M0.2;
5. simulation runtime starts paused at step zero;
6. pause/resume/single-step behave deterministically;
7. all six speed multipliers execute the expected integer fixed-step count;
8. identical total elapsed time produces the same result despite different pulse segmentation;
9. queued commands execute FIFO only at the next fixed-step boundary;
10. architecture tests reject wall-clock or timer APIs inside the Simulation project.
