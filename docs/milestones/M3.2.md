# M3.2 — Deterministic Multi-Component Network Orchestration

## Status

BASELINE CANDIDATE

Built on validated M3.1 — Plant Composition & Topology Baseline.

## Goal

Execute the canonical M3.1 plant topology as one deterministic staged network step without introducing new physical equations.

## Delivered

- `PlantNetworkOrchestrator`;
- committed-state-only component solving;
- deterministic accumulation of pipe, valve, pump, heat-transfer and heat-source balances;
- exactly-once integration of every fluid node and thermal body;
- `PlantNetworkStepResult` with immutable candidate state and canonical balance diagnostics;
- `PlantNetworkAudit` for global mass/energy accounting;
- explicit accounting of pump hydraulic work and external heat-source power;
- order-independence tests across shuffled caller registries;
- parallel-connection test proving components read the same committed endpoint state;
- fixed-step runtime pulse-segmentation integration test.

## Non-goals

- no new hydraulic/thermal equations;
- no core zones or fuel-channel groups;
- no steam drums or separator physics;
- no mass source/sink boundaries;
- no turbine/secondary cycle;
- no automatic plant control.

## Validation commands

```powershell
dotnet restore
dotnet build --no-restore
dotnet test --no-build
```

## Acceptance criteria

M3.2 is validated when:

- the complete solution builds with warnings as errors;
- the complete test suite passes;
- parallel components demonstrably solve against one committed state;
- changing caller registration order does not change candidate physical state;
- each conserved inventory is integrated once per fixed step;
- plant-level mass/energy audit residuals are bounded by test tolerances;
- runtime pulse segmentation produces identical final snapshots.
