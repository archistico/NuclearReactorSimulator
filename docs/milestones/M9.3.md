# M9.3 — Advanced Xenon & Low-Power Transients

## Goal

Promote the already validated M2.8 iodine/xenon model through the integrated operational runtime and use that canonical state for deterministic restart-after-shutdown and low-power poisoning scenarios.

## Status

**VALIDATED.**

The user explicitly confirmed successful local compilation and the complete automated test suite after the two test-only hotfixes. **M9.3 — Advanced Xenon & Low-Power Transients** is validated; M9.4 and M9.5 were subsequently validated; M9.5 is now the current validated functional baseline for the M9.6 candidate.

## Included

- optional `IodineXenonDefinition` composition in `ReactorPrimaryControlSystemDefinition`;
- canonical `IodineXenonState` carried by `ReactorPrimaryControlState`;
- M2.8 `IodineXenonSolver` invoked inside the M5.3 reactor/primary step when that definition is configured;
- committed xenon reactivity composed explicitly with the existing non-rod reactivity seam before point kinetics;
- candidate I/Xe inventory evolution from the resulting fission power and neutron population;
- immutable committed/candidate iodine-xenon diagnostics in `ReactorPrimaryControlSnapshot`;
- canonical xenon reactivity promoted by `ControlRoomSnapshotProjector` when the runtime configuration owns M2.8 poison state;
- versioned `XenonRestartInitialConditionFactory` and `LowPowerXenonInitialConditionFactory`;
- `AdvancedXenonScenarioPack` with restart-after-shutdown and low-power poisoning challenges;
- deterministic regression coverage for xenon reactivity composition, poison-state advancement, scenario reconstruction and history-dependent xenon evolution.

## Compatibility and versioning rule

Existing M7 initial-condition version 1 factories are not silently changed into xenon-enabled seeds. Their validated behavior remains unchanged and their xenon presentation remains explicitly unavailable.

M9.3 enables canonical iodine/xenon integration only for new versioned M9.3 initial conditions that carry explicit poison inventory state. This preserves exact initial-condition identity semantics for M9.1 replay/checkpoints while allowing new scenarios to expose the validated M2.8 phenomenon.

## Canonical step order

```text
committed rods + committed point kinetics + committed I/Xe
        ↓
committed rod reactivity
committed fission power → committed xenon reactivity
        ↓
external non-rod reactivity + xenon reactivity
        ↓
total reactivity → M2 point kinetics
        ↓
candidate neutron population → candidate fission power
        ↓
M2.8 IodineXenonSolver finite-step update
        ↓
candidate I/Xe state
```

The scenario/Application/UI layers do not integrate poison inventories and do not prescribe xenon curves.

## Scenario boundary

### Restart After Shutdown — Xenon Challenge

- exact versioned post-shutdown poison inventory seed;
- residual source-range/low-power kinetics seed;
- main circulation established;
- turbine and grid isolated;
- operator uses canonical rods and protection seams only;
- recovery timing/outcome is not scripted.

### Low-Power Xenon Manoeuvring Challenge

- exact versioned low-power poison-memory seed;
- canonical xenon reactivity is observable;
- operator manoeuvres with rods/HOLD and observes period/reactivity;
- no target xenon concentration, target recovery time or forced power trajectory.

## Deliberate limitations

M9.3 does not add:

- a second iodine/xenon solver in Application or scenario code;
- spatial xenon oscillations or per-zone poison inventories;
- samarium or depletion/burnup physics;
- plant-specific RBMK isotope constants or licensing-grade depletion fidelity;
- scripted xenon pits, recovery curves or predetermined restart success;
- hidden true-state fallback when the runtime has no configured canonical poison state.

## Validation result

M9.3 passed local compilation and the complete automated suite after two narrowly scoped test-only hotfixes:

1. added the missing M2.8 iodine/xenon namespace import in a Simulation test;
2. bounded an Application-level end-to-end xenon seed regression so it verified wiring/determinism without accidentally becoming an unrelated long-horizon M3 drum-envelope endurance test.

Neither hotfix changed production physics, scenario/initial-condition semantics, replay/versioning contracts or the validated M2.8 ownership model. M9.3 therefore became the official validated baseline for subsequent work and was later superseded by validated M9.4.
